# Impeccable Design Context — Telegraph Hub

This file defines the persistent design context for **Telegraph Hub** — a
single-binary Rust + Axum + HTMX web tool that proxies the Telegraph API
through a server-side-rendered UI. All `impeccable:*` skills and future
design sessions should read this first before making design decisions.

**Style name:** Terminal-flavored Dark Minimalism
**Established:** 2026-04-06

---

## Design Context

### Users

The primary (and currently only) user is a solo developer managing 7000+
Telegraph pages across multiple accounts. The tool replaces a Postman-based
workflow that broke at scale. Usage context is a desktop browser in long
sessions, doing high-density data operations: searching, batch-deleting,
previewing content, editing. The user is engineering-literate,
keyboard-oriented, and allergic to any UI that gets in the way of the task.

The job-to-be-done is *throughput*: burn through thousands of pages,
filter them, preview content inline, delete in bulk, and move on. Each
interaction should feel like another command in a terminal session — fast,
clear, no ceremony.

Hardware constraint is real: the tool must run smoothly on aging hardware.
Ten-year-old Android phones, old Intel integrated graphics, underpowered
laptops — these are not edge cases to tolerate, they are part of the core
target. Every design choice is filtered through "does this cost frames on
an old machine?"

### Brand Personality

**Focused. Utilitarian. Engineer-like.**

The voice sits closer to `htop` / `btop` / `man page` than to any consumer
SaaS landing page. The tool should feel like a well-engineered CLI wearing
just enough HTML/CSS to be clickable. Nothing exists to impress — every
pixel earns its place by doing a concrete job.

Emotional target: **Efficient momentum.** Actions should chain together
without friction — type a search, glance at results, hit a key, move on.
No drag, no waiting on decorative transitions, no visual noise between
thought and action. The interface should feel like typing commands in
sequence at a shell prompt.

### Aesthetic Direction

Dark-native minimalism with a terminal/htop flavor. Near-black background,
near-white text, a single cool accent used like a highlighter rather than
decoration, semi-transparent white borders instead of solid dark lines,
zero shadows, zero gradients, zero filters, system fonts only.

**Positive references** (patterns to borrow):
- **xAI** (`x.ai/DESIGN.md`) — near-black background with subtle cool
  undertone, monospace-as-voice, sharp corners, hover dims instead of
  brightens, absence as signal, border opacity as depth
- **Ollama** (`ollama/DESIGN.md`) — system-font-only pragmatism, binary
  radius philosophy, content restraint, zero-shadow discipline
- **htop / btop / Linear issue list** — dense data tables without
  decoration, row hover as the only separation, tabular numerics,
  uppercase monospace column headers, information density as identity

**Anti-references** (patterns to avoid):
- Linear, Vercel, Mintlify — the "new startup SaaS" aesthetic with
  atmospheric gradients, custom web fonts, `backdrop-filter: blur()`,
  and emotional hero sections
- Material Design / Bootstrap card shadows — every shadow is a
  performance tax on old hardware
- Any design that depends on downloaded fonts, gradient meshes,
  `backdrop-filter`, custom animated SVGs, or heavy keyframe animation
- Consumer app warmth — rounded corners above 4px, pastel backgrounds,
  decorative illustrations, emoji in UI chrome

### Design Principles

Five principles guide every decision in this project:

1. **Performance before aesthetics.** If a visual choice would cause
   jank on a ten-year-old Android or an Intel HD 4000 laptop, it is
   wrong. Zero `box-shadow`, zero gradients, zero `backdrop-filter`,
   zero font downloads. Animations limited to `opacity` and
   `background-color` with ≤160ms duration. `prefers-reduced-motion`
   honored absolutely.

2. **Absence as signal.** When something needs visual weight, remove
   something else instead of adding. A thinner border, a different
   background opacity step, a column of whitespace — these are stronger
   than a shadow or a color fill. The interface is a relief carved out
   of dark, not a collection of floating elements.

3. **One accent, used as a tool.** `#9ab5ff` (cool blue-violet) appears
   only where interaction or state demands attention: primary button,
   active pagination page, focus ring, link. Never decoratively. Never
   as a background fill except as `rgba(154,181,255,0.14)` tint for
   selected rows.

4. **Density is a feature.** This is not a marketing page; it's a
   7000-row table. Tabular numerics on every numeric column, monospace
   for paths and tokens, 14px body text, 12–13px for metadata,
   line-height 1.5 on table cells. Engineers want to see more, not less.

5. **WCAG AAA or nothing.** All text-on-background pairs must reach 7:1
   contrast. Focus indicators must be visible without being decorative.
   `prefers-reduced-motion` and `prefers-color-scheme: light` both fully
   supported. No exceptions for readable content. Disabled states
   deliberately below AAA threshold because they must not read as
   active.

## Design System

### Color Tokens

All values are WCAG AAA verified against `--bg` (`#1a1c22`).
Contrast ratios in parentheses.

#### Background (luminance stacking, not shadows)
```
--bg               #1a1c22                   page canvas
--bg-1             rgba(255,255,255,0.04)    subtle surface (cards, table)
--bg-2             rgba(255,255,255,0.07)    hover surface
--bg-3             rgba(255,255,255,0.10)    active / pressed surface
```

`#1a1c22` is a near-black with a subtle cool blue undertone (R=26, G=28,
B=34). It avoids both the harsh eye strain of pure `#000000` and the
neutral flatness of `#121212`. Same family as xAI's `#1f2228`.

#### Text (all AAA compliant)
```
--text             #f5f6f8                    primary    (15.76:1)
--text-2           rgba(245,246,248,0.85)     secondary  (11.59:1)
--text-3           rgba(245,246,248,0.68)     muted      ( 7.85:1)
--text-disabled    rgba(245,246,248,0.38)     disabled   (below AAA by design)
```

Note: `--text-3` uses 0.68 alpha specifically to clear the AAA 7:1 bar.
Anything lower (0.50, 0.55) fails AAA for normal text and must not be
used for readable content.

#### Border (semi-transparent white, never solid dark)
```
--border           rgba(255,255,255,0.10)    default
--border-strong    rgba(255,255,255,0.18)    hover / active
--border-subtle    rgba(255,255,255,0.06)    barely-there dividers
```

Solid dark borders on dark backgrounds are forbidden — they either
disappear or read as cracks. Semi-transparent white is the only
border treatment in this system.

#### Accent (single cool blue-violet)
```
--accent           #9ab5ff                   primary  ( 8.45:1)
--accent-hover     #b0c5ff                   hover    (~10.5:1)
--accent-soft      rgba(154,181,255,0.14)    tinted bg for selected rows
```

Note: raised from the exploration's initial `#7a9eff` (which only reached
6.65:1, failing AAA) to `#9ab5ff`. Hue preserved, lightness bumped. Any
future tweak to this value must verify 7:1 contrast against `--bg`.

#### Semantic
```
--danger           #ff8a8a                   destructive  ( 7.50:1)
--danger-hover     #ffa0a0                   hover
--danger-soft      rgba(255,138,138,0.12)    error bg

--success          #7adba4                   success      (10.14:1)
--success-soft     rgba(122,219,164,0.12)    success bg
```

No warning/info colors — semantic palette is intentionally minimal.

### Typography

System fonts only. No `@font-face`. No downloads. No FOUC.

```
--font-sans   -apple-system, BlinkMacSystemFont, "Segoe UI", Roboto,
              Helvetica, Arial, sans-serif
--font-mono   ui-monospace, "Cascadia Code", "SF Mono", Menlo, Consolas,
              monospace
```

#### Weights (three only)
- `400` — body, reading text
- `500` — UI, emphasis, navigation, button labels
- `600` — headings, page titles
- never `300`, never `700`

#### Scale
| Token | Size | Use |
|-------|------|-----|
| `--fs-xs` | 12px | micro label, monospace tag, column header |
| `--fs-sm` | 13px | caption, metadata, pagination |
| `--fs-base` | 14px | table cell, button label, body UI |
| `--fs-md` | 16px | form input, paragraph body, preview body |
| `--fs-lg` | 20px | section heading, card title |
| `--fs-xl` | 28px | page title |

#### Line-height
| Token | Value | Use |
|-------|-------|-----|
| `--lh-tight` | 1.1 | page titles |
| `--lh-snug` | 1.3 | section headings |
| `--lh-base` | 1.5 | body, table cells, UI |
| `--lh-relaxed` | 1.65 | long-form content (preview panel) |

#### Monospace usage
Monospace is reserved for: paths, tokens, URLs, IDs, numeric counts,
status codes, column headers (uppercase), and any inline technical
reference. It is part of the visual voice — it says "this is structured
data, not prose."

#### Tabular numerics
All numeric columns, pagination numbers, view counts, and batch counters
must use `font-variant-numeric: tabular-nums` so digits align vertically
in a scrolling table. This is non-negotiable.

### Radius (three-tier binary system)
```
--radius-sm     2px      buttons, inputs, pagination controls
--radius-md     4px      cards, table, preview panel
--radius-full   9999px   chips, badges, status dots only
```

Nothing in between. The jump from 4px to 9999px is intentional — it
creates a clear binary between "structural container" and "status marker".

### Spacing (4px base, 8px rhythm)
```
--space-1   4px
--space-2   8px
--space-3   12px
--space-4   16px
--space-5   24px
--space-6   32px
--space-7   48px
--space-8   64px
```

### Motion

```
--transition-fast   90ms  ease
--transition-base   120ms ease
--transition-slow   160ms ease
```

**Allowed properties only:**
- `opacity`
- `background-color`
- `color`
- `border-color`

**Forbidden properties:**
- `transform` (forces compositor layer, jank on old GPUs)
- `filter` / `backdrop-filter` (extreme performance cost)
- `box-shadow` (there are no shadows anyway)
- `width` / `height` (forces layout reflow)

**Required reduced-motion fallback:**
```css
@media (prefers-reduced-motion: reduce) {
  *, *::before, *::after {
    transition: none !important;
    animation: none !important;
  }
}
```

**Existing animations to simplify:**
| Current animation | Target |
|-------------------|--------|
| `panelSlideIn` (`translateX(20px) → 0`) | fade only (`opacity 0 → 1`) |
| `slideUp` (`translateY(8px) → 0`) | fade only |
| `slideIn` (`translateX(100%) → 0`) | fade only |
| `spin` | keep unchanged (functional loading indicator) |

### Elevation (via opacity, never via shadow)
```
Level 0    --bg                               page background
Level 1    --bg-1 + --border                  resting cards, table
Level 2    --bg-2                             hover state
Level 3    --bg-3 + --border-strong           active / pressed / focused
```

No `box-shadow` exists in this design system. Depth is expressed only
through (a) background luminance steps and (b) border opacity steps.

### Focus Ring (AAA-strong)

Focus indicators must be visible at AAA standard and must not rely on
color alone.

```css
*:focus-visible {
  outline: 2px solid var(--accent);
  outline-offset: 2px;
}
```

Every interactive element (buttons, links, inputs, table row checkboxes,
pagination buttons, select elements) must have a visible focus ring.
Keyboard navigation through the entire interface must be possible and
traceable. Never use `outline: none` without providing a replacement.

### Data Table Style (htop-flavored)

The page listing is the most important screen in this application. Its
table follows htop/btop conventions rather than spreadsheet conventions.

- No outer border, no vertical column lines
- Column headers: uppercase, 12px monospace, letter-spacing 1px,
  `--text-3` color, bottom border `--border`
- Rows: no border between rows by default
- Row hover: `--bg-1` background (`rgba(255,255,255,0.04)`)
- Row selected (checkbox checked): `--accent-soft` background
- Row deleted (placeholder `[DELETED]`): opacity 0.35, no actions
- Zebra striping: **off** — terminal aesthetic, not spreadsheet
- Pagination: monospace numbers, `--radius-sm`, active page uses
  `--accent` background with `--bg` text
- Path column: `--font-mono`, `--text-2`, hover reveals external link
  icon on the right
- Numeric columns (views, count): `tabular-nums`, right-aligned

### Light Mode (fallback only, not default)

Light mode exists for users with `prefers-color-scheme: light` explicitly
set. Dark is the default. Light mode is a structural inversion of the
dark system, not a separate design.

```
--bg               #fafbfc
--text             #1a1c22     (inverted)
--border           rgba(0,0,0,0.10)
--accent           #2e4db0     (AAA ≈ 7.3:1 against #fafbfc)
```

All AAA contrast ratios must be re-verified when implementing light
mode. The dark values do not translate by simple inversion for
semantic colors.

## Do / Don't

### Do
- Use `#1a1c22` for dark mode background, `#fafbfc` for light mode
- Use `#9ab5ff` as the single accent, applied sparingly
- Use system fonts only — no `@font-face`, no font downloads
- Use `tabular-nums` on every numeric column
- Use monospace for paths, tokens, IDs, and column headers
- Use semi-transparent white borders (`rgba(255,255,255,0.10)`) on dark
- Use background opacity stepping for elevation
- Honor `prefers-reduced-motion` on every animated element
- Maintain 7:1 contrast on all readable text (WCAG AAA)
- Use `outline: 2px solid var(--accent)` with `outline-offset: 2px`
  for focus rings
- Keep transitions ≤160ms
- Use the binary radius system: 2px / 4px / 9999px

### Don't
- Don't use `box-shadow`, ever
- Don't use `linear-gradient`, `radial-gradient`, or `conic-gradient`
- Don't use `backdrop-filter` (huge performance cost)
- Don't use `filter` for hover effects
- Don't use `transform` for decorative animations (only for spin)
- Don't download web fonts (Inter, Geist, SF Pro, JetBrains Mono, etc.)
- Don't use solid dark colors for borders on dark backgrounds
- Don't use `color-mix()` (old Safari compatibility)
- Don't use radius values between 4px and 9999px (except 2px)
- Don't use font weights 300, 700, 800, or 900
- Don't use pure black `#000000` or pure white `#ffffff`
- Don't add decorative hover animations — hover indicates interactivity,
  not delight
- Don't use chromatic colors beyond the accent, danger, and success
  tokens
- Don't use zebra striping on tables
- Don't ship any text below 7:1 contrast if it needs to be read
- Don't use `outline: none` without providing a replacement

## File Impact Scope

The redesign touches only the presentation layer. Rust code, Telegraph
API client, cache logic, i18n machinery, HTMX behavior, and route
structure are all out of scope.

| File | Change |
|------|--------|
| `static/style.css` | Full rewrite (842 → ~600 lines target) |
| `templates/base.html` | Theme switch default flipped to `dark` |
| `static/app.js` | `toggleTheme()` default value inverted |
| `templates/page_list.html` | Table structure aligned to data-table style |
| `templates/fragments/page_row.html` | Monospace path, tabular views |
| `templates/fragments/page_preview.html` | Preview body typography reset |
| `templates/fragments/toast.html` | `slideIn` transform → fade |
| `templates/fragments/account_card.html` | Card refinements |
| `templates/fragments/search_progress.html` | Spin indicator review |
| `templates/index.html` | Card refinements |
| `templates/page_editor.html` | Form styling refinements |
| `templates/account_info.html` | Info list refinements |

**NOT touched:**
- `src/` — all Rust code, route handlers, Telegraph client, cache, i18n
- `locales/*.json` — translation content
- `Cargo.toml` / dependencies
- HTMX behavior, route paths, server logic, database schema

## Build Order (impeccable skill chain)

The redesign is staged through the `impeccable:*` skill chain, each
step reading this file as its design context.

1. `impeccable:teach-impeccable` — writes this file (this step)
2. Create OpenSpec change `redesign-ui-terminal-dark` with
   proposal.md, design.md, and tasks.md
3. `impeccable:audit` — baseline technical check of current state
   (accessibility, performance, theming, responsive, anti-patterns)
4. `impeccable:critique` — baseline UX assessment of current state
5. `impeccable:distill` — strip `style.css` of decorative complexity
6. `impeccable:typeset` — reset typography scale and hierarchy
7. `impeccable:colorize` — apply the AAA-compliant dark palette
8. `impeccable:arrange` — reset spacing and layout rhythm
9. `impeccable:extract` — consolidate CSS variables into design tokens
10. `impeccable:animate` — replace transforms with fades, 120ms limit
11. `impeccable:adapt` — verify responsive behavior across breakpoints
12. `impeccable:harden` — handle i18n overflow, edge cases, error states
13. `impeccable:normalize` — align any stragglers with the design system
14. `impeccable:polish` — final micro-detail pass
15. `impeccable:audit` — verification against baseline report

## Agent Prompt Guide

When writing CSS or templates for this project, default to these values
without asking:

- Background: `#1a1c22` (dark) / `#fafbfc` (light)
- Primary text: `#f5f6f8` / `#1a1c22`
- Accent: `#9ab5ff` / `#2e4db0`
- Border: `rgba(255,255,255,0.10)` / `rgba(0,0,0,0.10)`
- Radius: `2px` for interactive, `4px` for containers, `9999px` for chips
- Font: system stack only
- Transition: `120ms ease`, opacity/background-color only
- Shadow: none
- Gradient: none
- Focus ring: `2px solid var(--accent)` with `2px` offset

When in doubt, remove instead of add. Absence is the default.
